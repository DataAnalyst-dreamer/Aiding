# Business Requirements Document (BRD) - 아이딩(Aiding) 앱

## 1. 프로젝트 개요 (Executive Summary)
**아이딩(Aiding)**은 초보 부모의 육아용품 결정 장애를 해결하기 위해 기획된 **AI 기반 월령 맞춤 육아용품 큐레이터 앱**입니다. 
본 문서는 기존에 진행된 MVP 코드베이스 리뷰 결과와, 사업 기획을 위해 진행된 전문가 토론 결과 및 1인 부업 수익화 기획안(최종 도출안)을 통합하여 작성되었습니다. 

본 프로젝트의 궁극적인 비즈니스 목표는 **Antigravity AI Assistant**를 코-파일럿으로 활용하여 비개발자가 8주 내에 MVP를 완성하고, 정교하게 설계된 비즈니스 모델(Freemium 구독 + 제휴 수수료)을 통해 출시 6개월 내 구독자 250명과 **월 수익 100만 원 이상을 달성**하는 것입니다.

## 2. 앱 정체성 및 타겟 고객 (Identity & Target Audience)

### 2.1. 앱 정체성 (App Identity)
- **앱 이름:** 아이딩 (Aiding) - 'AI' + '아이(Child)' + '돕다(Aiding)'의 삼중 의미
- **핵심 가치 (Core Value):** 초보 부모의 육아용품 결정 장애를 AI로 해결
- **포지셔닝:** 정보 앱과 커머스 앱의 교차점 역할을 하는 "육아용품 AI 큐레이터"
- **차별화 포인트:** 월령 자녀 정보 기반 자동 갱신 AI 추천 + 24시간 육아 Q&A + 큐레이션 코멘트

### 2.2. 타겟 고객
- **주 타겟:** 0~36개월 첫째 아이를 둔 30대 초반 맞벌이 부모
- **특징 및 니즈:**
  - 육아용품의 성분, 리뷰, 가격 등을 꼼꼼하게 비교하지만 정보 탐색 시간이 현저히 부족함.
  - 내 아이의 현재 발달(월령) 상황에 적합한 '신뢰할 수 있고 구체적인 아이템 추천'을 원함.

## 4. 기능적 요구사항 (Functional Requirements)

### 4.1. 사용자 및 인증 관리
- **F-REQ-01 (소셜 로그인):** 구글/애플 로그인 기능 제공 (Firebase Auth 또는 Supabase Auth 연동).
- **F-REQ-02 (온보딩 및 자녀 정보):** 최초 진입 시 필수적으로 아이의 생년월일(월령 계산용)을 입력받아 NoSQL/SQL DB에 저장.

### 4.2. 메인 화면 및 큐레이션 (홈·추천관 탭)
- **F-REQ-03 (UI/UX 디자인 가이드):** 하단 4개 탭(홈/추천관/AI상담/성장리포트) 구조. 메인 컬러 민트그린(#4ECDC4), 웜화이트(#FAFAF8) 배경, 코랄(#FF6B6B) 포인트 색상 적용.
- **F-REQ-04 (월령 맞춤 필터링 카드):** 현재 아이의 월령에 적합한 (`minMonth` ~ `maxMonth`) 육아용품 리스트 노출.
- **F-REQ-05 (추천 상품 DB 데이터 구조):** 제휴 링크(naverUrl/coupangUrl), 실시간 가격, 클릭 수, Gemini가 생성한 전문용 큐레이터 코멘트(`aiDescription`), 프리미엄 여부(`isPremium`) 항목 유지.
- **F-REQ-06 (컨텐츠 잠금):** 프리미엄 전용 상품에 대해서는 흐릿한 블러 처리 및 자물쇠 아이콘으로 결제 단계를 유도.

### 4.3. AI 육아 Q&A 및 발달 리포트 (상담·리포트 탭)
- **F-REQ-07 (자연어 육아 상담):** Gemini API 바인딩을 통해 24시간 아이 발달 육아 상담 제공. 단, 답변 시 "전문 의료 조언이 아님"을 명시하는 면책조항 필수 포함.
- **F-REQ-08 (사용 제한 카운팅):** 무료 사용자의 경우 월 3회로 AI Q&A 상담 횟수를 제한하여 서버사이드 로직 필드 내에 제어 로직 수립.
- **F-REQ-09 (발달 마일스톤 및 성취 리포트):** 리텐션을 위해 AI가 작성하는 '성장 리포트 창출 기능'을 도입하고, 이를 외부로 공유할 수 있는 버튼(추천인 코드 포함) 배치.

### 4.4. UI/UX 및 사용자 경험 시나리오 (UI/UX User Flow)
비개발자 1인 창업의 특성상 빠른 개발을 위해, 사용자가 앱을 켜서 결제하기까지의 흐름(User Flow)과 예외 상황을 다음과 같이 아주 구체적으로 분리하여 설계합니다.

- **F-REQ-10 (핵심 유저 흐름 - User Flow):**
  1. **[회원가입/로그인 화면]**: "구글로 3초 만에 시작하기" 버튼 노출.
  2. **[온보딩 화면]**: "아이의 생년월일이 언제인가요?"를 묻는 귀여운 달력 선택기 제공. (이 정보로 모든 AI 추천이 개인화됨)
  3. **[메인 홈 화면]**: 상단에 "ㅇㅇ이가 태어난 지 150일째예요!" 환영 문구 제공. 하단에는 지금 월령에 꼭 필요한 필수 육아템 베스트 3 노출.
  4. **[AI 상담 화면]**: 카카오톡처럼 편안한 채팅창. 첫 진입 시 AI가 먼저 "요즘 밤잠은 잘 자나요?" 등 친근하게 말 걸기.
  5. **[결제 유도(Paywall) 화면]**: 무료 횟수(3회)를 다 쓰거나 프리미엄 상품을 누르면, 화면이 반투명해지면서 자물쇠 아이콘과 함께 "월 2,900원으로 ㅇㅇ이 맞춤 프리미엄 추천을 무제한 받아보세요 (7일 무료 체험)" 팝업 노출.

- **F-REQ-11 (데이터가 없을 때의 화면 - Empty State UX):**
  - **상황:** 사용자가 첫 가입 후 찜한 상품이 없거나, AI 추천 기록이 없을 때.
  - **대응:** 텅 빈 하얀 화면 대신, 귀여운 아기 울음 일러스트와 함께 "아직 추천받은 아이템이 없어요. AI 큐레이터에게 첫 질문을 남겨보세요!"라는 안내 문구와 [지금 질문하러 가기] 버튼 배치.

- **F-REQ-12 (오류 발생 시 화면 - Error Handling UX):**
  - **상황:** 인터넷이 끊기거나 AI 서버 응답이 지연될 때.
  - **대응:** 앱이 멈추거나 영어로 된 에러 메시지를 띄우지 않고, 화면 하단에 검은색 알림창(SnackBar)으로 "앗, 통신이 잠시 원활하지 않아요. 다시 시도해 주세요 🥲"와 직관적인 [재시도 버튼] 도출.

## 5. 비기능적 요구사항 및 인프라 (Non-Functional Requirements)

### 5.1. 기술 스택 및 개발 원칙
- **개발 환경:** **Flutter** (크로스 플랫폼 모바일 프레임워크 유지 권장)
- **AI 개발 파트너:** **Antigravity (Google DeepMind Agent)**를 메인 코딩 엔진으로 활용하여 아키텍처, 코드 작성, 디버깅 전 과정을 수행
- **백엔드/데이터베이스 (재평가 결과):**
  - **옵션 A (기존 최적화): Firebase Auth / Firestore / Cloud Functions** - 비개발자의 관리 용이성, NoSQL의 유연함, No-Ops 운영 측면에서 여전히 매우 훌륭한 선택지. Antigravity가 Firebase Functions (Node.js) 관련 코드 작성도 원활히 지원함.
  - **옵션 B (SQL 및 비용 절감 대안): Supabase (Auth / PostgreSQL / Edge Functions)** - 장기적으로 관계형 데이터베이스(SQL)가 필요하거나 Firebase의 종량제 비용이 부담될 때 훌륭한 대안.
  - *결론: MVP 단계(1인 부업, 주 6시간 운영 모델)에서는 **기존 Flutter + Firebase 조합을 유지하고, Antigravity를 이용해 모바일-서버 간 로직을 분리 구축(Cloud Functions 활용)하는 것을 강력히 권장**함.*
- **AI 모델:** Google Gemini API 
- **구독 결제 모듈:** RevenueCat 연동을 통한 인앱결제/구독 상태 동기화 관리
- **N-REQ-01 (API 보안 최우선 개선):** 앱 클라이언트 소스 내 `GEMINI_API_KEY` 하드코딩 탈피 → Firebase Cloud Functions 등 중간 백엔드로 호출 이관을 Antigravity를 통해 구축.
- **N-REQ-02 (상태 캐싱):** 화면 이동마다 DB 재호출 방지를 위한 Provider 상태 관리(Repository 패턴) 아키텍처 고도화 반영.

### 5.2. 운영 및 법적 대응
- **N-REQ-03 (개인정보 및 약관):** 출시 심사를 피하기 위해 앱 내부 및 스토어에 AI로 초안을 작성한 '개인정보처리방침' 및 '이용약관(면책조항 포함)' 필수 등록 및 연결.
- **N-REQ-04 (보안 규칙 보강):** 인증된 사용자만 자신의 개인정보/채팅 데이터에 접근할 수 있도록 보안 규칙 셋업.
- **N-REQ-05 (1인 주간 운영 루틴 반영):** 주 평균 6시간(버그 수정 1h, 컨텐츠 생성/포스팅 2h, 지표 모니터링 및 DB업데이트 2h, 커뮤니티 답변 1h) 범위 내에서 유지보수가 가능하도록 모든 기술적 구조는 단순성 및 자동화(상품 DB API 수집 봇 등)를 향해야 함.

## 7. 실행 액션 플랜 및 다음 과제

현재 **개발 전 1주차** 단계로 고려하였을 때, 가장 시급히 처리되어야 할 액션 아이템은 다음과 같습니다.
1. 상표권(키프리스) 및 앱스토어 이름 중복 체크 완료 후 프로젝트 셋업 (`아이딩 / Aiding`)
2. Firebase 프로젝트 준비, 클라우드 함수(Functions) 연동을 위한 Antigravity 환경 동기화
3. 제휴 수익 창출 기반인 '네이버 개발자센터 API' 및 '쿠팡파트너스' 가입
4. 기 작성된 초기 버전의 앱 코드를 기반으로, **'수익화 결제 로직(RevenueCat)' 및 '공유하기/리포트' 시스템 확장** 본격 돌입.
